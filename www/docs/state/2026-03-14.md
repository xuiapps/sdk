# Xui Framework State Evaluation

**Date:** March 14, 2026
**Author:** AI-generated analysis (Claude, Anthropic) with author commentary
**Benchmarked against:** .NET MAUI, Flutter, Avalonia UI, React Native

> **From the Xui author:** The analysis below recommends a plan that would align Xui with existing frameworks — filling in the standard widget catalog, matching feature parity, and so on. That's a valid long-term direction, but it's not the immediate priority. In the coming months we will focus on delivering on Xui's **differentiators** — the things that make it unlike anything else — rather than trying to build a competitor or a copy-cat. The unique rendering architecture, zero-allocation design, and developer tooling (emulator, devtools, software renderer) are what justify Xui's existence. We'll build the standard toolkit pieces over time, but the near-term work is about making the foundation undeniably excellent.

---

## Why Xui Exists

Most cross-platform UI frameworks make a trade-off: either wrap native controls (losing consistency and control) or ship a third-party rendering engine (adding size, complexity, and an opaque dependency). Xui takes a different path: it draws directly using each platform's **native graphics API** — Direct2D on Windows, Core Graphics on macOS/iOS, Canvas on Android, Canvas 2D on Browser — with no Skia, no Impeller, no WebView.

### Advantages

- **No third-party rasterizer.** Xui talks directly to the OS graphics stack. There is no Skia binary to ship, update, or debug through. Each platform uses the GPU pipeline the OS already optimizes for.
- **Zero-allocation event system.** Events are ref structs passed by reference. The input hot path produces no GC pressure — measurably better than any other .NET UI framework.
- **Composable layers.** Visual effects (borders, focus rings, text input decoration) are zero-alloc `ILayer` structs composed via generics, not nested heap-allocated wrapper widgets.
- **Tiny footprint.** Xui adds only its managed code on top of the shared .NET runtime. No bundled engine, no second runtime, no embedded browser.
- **Pixel-perfect testing.** The Software renderer produces deterministic output without a GPU, enabling screenshot-based regression tests in CI.
- **Phone emulator middleware.** Test mobile layouts (notch, safe areas, form factors) on desktop without deploying to a device.
- **.NET 10 / C# 13.** Full access to the latest language features, NativeAOT, `Span<T>`, ref structs, source generators.

### Trade-offs

- **Young framework.** Xui.Core has a solid rendering and layout foundation, but the widget library is minimal (6 widgets today). Building a real app requires contributing to the ToolKit effort.
- **No native control wrapping.** Xui draws everything itself. This means controls don't automatically match the latest OS design refresh — they need explicit theme work.
- **Small ecosystem.** No third-party packages, limited documentation, no Stack Overflow presence yet. Early adopters need to be comfortable reading source code.
- **Platform maturity varies.** Windows and macOS backends are the most mature. Android currently uses Xamarin-era native bindings (Skia hidden behind Android Canvas) — replacing these is an active priority. Browser support is experimental, built on early .NET WASM.

## What Xui Has Today

### Rendering & Graphics — Mature

The 2D drawing pipeline is the most complete subsystem. Xui owns its rendering stack across all platforms — no Skia, no third-party rasterizer.

- Full canvas API: `IContext` aggregates `IPathDrawingContext`, `IRectDrawingContext`, `ITextDrawingContext`, `IImageDrawingContext`, `ITransformContext`, `IMeasureContext`
- Color, `LinearGradient`, `RadialGradient`, `ImagePattern`
- `Path2D`, `CornerRadius`, `AffineTransform`
- Font system: `Font`, `FontMetrics`, `FontWeight`, `FontStyle`, `FontStretch`
- `LineCap`, `LineJoin`, `FillMode`, `TextAlign`, `TextBaseline`
- **7 platform backends:**

| Platform | Rendering | Windowing | Maturity |
|----------|-----------|-----------|----------|
| Windows | Direct2D 1.3 + DirectWrite + Direct3D 11 | Win32 | Mature |
| macOS | Core Graphics + Core Text + Core Animation | AppKit | Mature |
| iOS | Core Graphics + Core Text | UIKit | Functional |
| Android | Android Canvas via Xamarin native bindings | Android Activity | Legacy bindings — needs rework |
| Browser | Canvas 2D (WASM) | DOM | Experimental (.NET WASM) |
| Software | CPU rasterizer with tessellation | Headless | Stable |
| Test | Mock contexts | Mock window | Stable |

> **Android note:** The Android backend currently relies on Xamarin-era native bindings. Internally Android hides its own use of Skia behind the Android Canvas API, so Xui draws through that — but the managed interop layer is legacy code that needs to be replaced with direct JNI or a cleaner binding approach. This is an active priority.
>
> **Browser note:** The Browser backend builds on early .NET WebAssembly support and should be considered experimental. It works for demos and development but is not production-hardened.

### Layout System — Functional

The layout pipeline follows a Measure → Arrange → Render cycle with dirty-flag invalidation. It supports both block and inline flow directions.

- `HorizontalStack`, `VerticalStack` — flow layouts with wrap
- `HorizontalUniformStack`, `VerticalUniformStack` — equal-size distribution
- `ScrollView` — scrollable container with fling physics (`ExponentialDecayCurve`)
- Constraints: `Margin`, `MinimumWidth`/`MaximumWidth`, `MinimumHeight`/`MaximumHeight`
- Alignment: `HorizontalAlignment`, `VerticalAlignment` (Left, Center, Right, Stretch)
- Internationalization: `WritingMode`, `Direction` (LTR/RTL), `FlowDirection`
- Invalidation: `InvalidateMeasure()`, `InvalidateArrange()`, `InvalidateRender()`

### Widgets — Minimal

Xui.Core ships 6 concrete view types. This is enough to render text, images, and collect single-line text input, but far short of what's needed for a real application.

| Widget | What it does |
|--------|-------------|
| `View` | Base class for all UI elements |
| `Label` | Single-line styled text display |
| `TextBox` | Single-line text input with selection, caret, and IME |
| `ImageView` | Image display with intrinsic size |
| `Border` | Background, border stroke, padding, corner radius |
| `ScrollView` | Scrollable container with scroll bars and fling |

### Composable Layers — Unique to Xui

Xui's layer architecture is a zero-allocation alternative to deeply nested wrapper widgets. Layers are `ILayer` structs composed via generics.

- `BorderLayer<TView, TChild>` — border + padding + corner radius
- `ContentLayer` — padding wrapper
- `FocusBorderLayer<TView, TChild>` — focus-aware border styling
- `TextInputLayer` — text editing (selection, caret, IME integration)
- `DockLayer` — alignment docking utilities

### Input & Events — Solid

Events are ref structs passed by reference — zero allocations in the hot path.

- `EventRouter` with enter/leave generation, pointer capture, multi-pointer tracking
- Event types: `MouseMoveEventRef`, `MouseDownEventRef`, `MouseUpEventRef`, `ScrollWheelEventRef`, `TouchEventRef`, `KeyEventRef`, `InsertTextEventRef`, `DeleteBackwardsEventRef`
- `PointerState` with position, pressure, tilt, twist, button state
- Velocity tracking for fling gesture detection
- `EventPhase`: Capturing, AtTarget, Bubbling

### Animation — Basic

Easing and curve infrastructure exists, integrated into the layout pipeline's Animate pass. No high-level animation API yet.

- `Easing.CubicBezier`, `Easing.PolynomialEasing`
- `ConstantDecelerationCurve`, `ExponentialDecayCurve`
- `CubicMotionCurve`, `QuadraticMotionCurve`
- Frame-based timing via `TimeSpan` delta

### Curves & Math — Complete

Extensive 2D curve and geometry library used internally by the rendering and layout systems.

- Bezier curves: `LinearBezier`, `QuadraticBezier`, `CubicBezier`
- Splines: `LinearSpline`, `QuadraticSpline`, `CubicSpline` (monotonic variants)
- `Arc`, `Arc3Point`, `Segment`
- Geometry: `Point`, `Size`, `Rect`, `Frame`, `Vector`, `AffineTransform`
- 1D: `Interval<T>`

### Infrastructure — Strong

- **DI:** `IServiceProvider` with `IDeviceInfo`, `IRuntime`, `IDispatcher`, `IImagePipeline`
- **Emulator middleware:** Phone/tablet simulation with device chrome, notch, safe areas
- **DevTools middleware:** JSON-RPC protocol, visual tree inspection, `SplicingContext` overlay
- **Hot reload:** Runtime support for code changes
- **Test runtime:** `TestPlatform`, `TestWindow`, `TestRunLoop` for GPU-free testing
- **CSS-like classes:** `ClassNameCollection` on views
- **Build:** .NET 10, C# 13, Nerdbank.GitVersioning, NuGet packaging
- **Projects:** 39 projects across framework, runtimes, middleware, apps, tests, and utilities

## Architecture Principles

These principles guide all ToolKit implementation decisions:

**Build on Core, don't fork it.** ToolKit is a layer above Core. Every widget inherits from `Xui.Core.UI.View` and uses the existing layout/render pipeline. Missing primitives get added to Core and consumed by ToolKit.

**Zero-allocation by default.** Follow Core's ref-based pattern. Events, layout, and render hot paths must not allocate. Use `struct`, `Span<T>`, `stackalloc`, and pooling.

**Accessibility is not an afterthought.** Every widget gets semantic labels and roles from day one. The accessibility tree is built alongside the visual tree, not bolted on later.

**Theme tokens, not hard-coded values.** No widget contains a literal color, font size, or spacing. Everything comes from theme tokens. This enables dark mode, brand customization, and platform-adaptive styling.

**Composition over inheritance.** Build complex widgets by composing simpler ones (Button = TapGesture + Border + Label). The existing Layer architecture supports this.

**Platform-adaptive, not platform-identical.** Controls feel native on each platform without wrapping native controls. A Button uses Cupertino-style highlight on iOS and Material-style ripple on Android, driven by theme variants.

**Test against the Test Runtime.** Every widget has unit tests using `Xui.Runtime.Test`. Layout, hit testing, state changes, and accessibility labels are all testable without a GPU.

## Open Questions

**Markup language?** Code-only is simpler but XAML attracts WPF/MAUI developers. C# source generators could offer a lightweight declarative syntax without a separate language.

**Default design language?** Material 3 is well-documented and familiar. Alternatively, design a unique "Xui" look and ship Material as a theme option.

**Data binding approach?** .NET MVVM (`INotifyPropertyChanged`) vs. signals/observables vs. source-generated reactive properties. Source generators could eliminate boilerplate while staying idiomatic C#.

**SVG subset?** Full SVG is enormous. Define a practical subset (paths, fills, transforms, gradients) covering icon use cases.

**Adaptive or uniform?** Single look everywhere (Flutter-style) or adapt to platform (MAUI-style)? Recommendation: unified base with platform-adaptive variants via theme.
